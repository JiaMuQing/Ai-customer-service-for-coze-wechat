# AI 客服管理系统 — 项目计划文档

## 一、技术选型

| 层级 | 技术 | 说明 |
|------|------|------|
| 后端 | Node.js + Nest.js | 调用扣子 API、企业微信回调、会话与任务队列 |
| 前端 | Vue 3 + Vite + Pinia | 管理后台，可搭配 Ant Design Vue / Element Plus |
| 数据库 | MySQL 8 | 群绑定、Bot 配置、会话记录等；后续可加 FULLTEXT 或 ES |
| AI 能力 | 扣子（Coze）API | 知识库管理、Bot 对话（同一 bot_id，多会话靠 conversation_id 区分） |
| 触达 | 企业微信 | 智能机器人（群内需 @ 才回调）、应用发消息到群 |

- **登录**：本期固定写死用户名/密码，不做权限体系与用户管理。

                                    ▼

## 三、数据设计要点

- **群 ↔ Bot 绑定**：`wecom_group_id`、`bot_id`（扣子）、启用状态等；同一技能即同一 `bot_id`。
- **会话与上下文**：按「企业微信群 ID + 用户 ID」或按会话 ID 存多轮；调扣子时传 `conversation_id` 或 `additional_messages` 做上下文，实现同群多用户、同一问题多轮区分。
- **知识库**：在扣子侧创建/更新，本系统做展示与同步状态；实际存储与检索在扣子。
- **登录**：本期无用户表，固定账号密码写死在配置或代码，不落库、不做权限与角色管理。

---

## 四、核心流程

1. **企业微信回调**：用户群内 @ 机器人或单聊发消息 → 企业微信推送到后端接收消息 URL → 解密、幂等处理。
2. **路由**：根据 `chatid`（群）/ 单聊标识 + `from.userid` 查群绑定表得 `bot_id`，命中同一 `bot_id`。
3. **扣子对话**：用 `conversation_id`（或历史消息）区分不同用户/不同会话，调用扣子 Bot 对话 API（如 `/v3/chat`），可选流式。
4. **回复**：将 Bot 返回内容通过企业微信「应用发消息到群」或智能机器人主动回复接口回发到对应群/单聊。
5. **会话记录**：可选落库（MySQL）做查询与简单报表，不要求本期完整管理端。

## 五、前期准备项

| 类型 | 准备内容 |
|------|----------|
| 扣子 | 注册扣子、创建并发布 Bot、配置知识库；生成 **PAT**（个人访问令牌），权限含 chat、知识库等；记录 **bot_id**。 |
| 企业微信 | 创建**自建应用**，获取 corp_id、corp_secret、agent_id；在「智能机器人」能力中创建/绑定**智能机器人**，配置**接收消息**的回调 URL、Token、EncodingAESKey。 |
| 环境 | 安装 Node 20+、MySQL 8；准备可被企业微信访问的 HTTPS 回调域名。 |
| 本期不准备 | 权限体系、用户/角色/菜单管理、多租户。 |

## 六、本期范围（固定账号、暂不做权限与管理）

- **登录**：固定写死用户名与密码，校验通过即视为已登录，不建用户表、不做权限与角色。
- **功能**：  
  - 后端：企业微信智能机器人回调接入、扣子 Bot 对话（同一 bot_id、conversation 区分上下文）、群 ↔ Bot 绑定、会话落库（可选）。  
  - 管理端：知识库展示/同步状态、Bot 配置、群列表与绑定管理、简单会话记录查看；**不做**：用户管理、权限管理、角色与菜单配置。
- **知识库**：在扣子侧管理；本系统仅做展示与同步状态，不实现完整知识库 CRUD 管理后台。
- **交付**：可运行的后端 + 管理前端，固定账号登录即可使用上述能力。

---

## 七、企业微信群内行为说明

- **群聊**：仅当用户** @ 智能机器人**时，企业微信才会把该条消息回调到你的接收消息 URL；群内其他未 @ 的消息不会推送，机器人无法据此自动回复。
- **单聊**：用户对机器人发消息（含引用消息等支持类型）即会触发回调。

因此当前方案下，群内为「@ 机器人 → 机器人回复」模式，无法做到「看到群里所有聊天并自动回复」。
